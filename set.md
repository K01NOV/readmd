# Практическая работа №3: Тестовые наборы (Test Suits)


## Способ 1: Группировка по функциональным модулям
*Этот метод позволяет тестировщику сфокусироваться на проверке конкретной части системы, в которой велась разработка.*

### 1. Сьют "Аутентификация и Доступ"
* **Описание:** Проверка всех входных точек в систему, безопасности авторизации и разграничения прав.
* **Характеристика:** Функциональное тестирование прав доступа.
* **ID кейсов:** TC001, TC002, TC003, TC021, TC027, TC035.

### 2. Сьют "Управление каталогом и Поиск"
* **Описание:** Проверка поиска товаров, фильтрации и корректности отображения карточек товаров.
* **Характеристика:** Тестирование удобства поиска и навигации (Discovery).
* **ID кейсов:** TC012, TC013, TC014.

### 3. Сьют "Корзина и Покупки"
* **Описание:** Проверка логики добавления/удаления товаров и работы с купонами.
* **Характеристика:** Тестирование критического пути продаж.
* **ID кейсов:** TC010, TC011, TC015.

### 4. Сьют "Работа с заказами"
* **Описание:** Полный цикл заказа: создание, отмена, просмотр истории и возврат.
* **Характеристика:** Тестирование логики жизненного цикла заказа.
* **ID кейсов:** TC007, TC008, TC009, TC018.

## Способ 2: Группировка по приоритету (Уровни тестирования)
*Этот метод помогает определить очередность выполнения тестов при сжатых сроках.*

### 5. Сьют "Smoke Test (Критический)"
* **Описание:** Набор самых важных тестов, проверяющих, что основные функции (вход, покупка, оплата) не «сломаны».
* **Характеристика:** Проверка базовой работоспособности.
* **ID кейсов:** TC001, TC004, TC007, TC016.

### 6. Сьют "Extended Regression (Средний приоритет)"
* **Описание:** Подробная проверка функций, которые важны для пользователя, но не блокируют использование системы.
* **Характеристика:** Регрессионное тестирование.
* **ID кейсов:** TC005, TC008, TC012, TC022, TC029.

### 7. Сьют "Minor & UI (Низкий приоритет)"
* **Описание:** Проверка второстепенных функций, логирования и косметических правок интерфейса.
* **Характеристика:** Вспомогательное тестирование.
* **ID кейсов:** TC009, TC013, TC019, TC025.

## Способ 3: Группировка по виду тестирования (Техническая)
*Метод разделяет тесты по их технической сути и требованиям к окружению.*

### 8. Сьют "Security & Validation"
* **Описание:** Проверка устойчивости системы к взлому, валидация данных и безопасность сессий.
* **Характеристика:** Тестирование безопасности (Security Testing).
* **ID кейсов:** TC021, TC033, TC035.

### 9. Сьют "System Performance & Stability"
* **Описание:** Тестирование работы API, логирования ошибок и поведения системы при обрыве сети.
* **Характеристика:** Нефункциональное тестирование (Reliability).
* **ID кейсов:** TC019, TC020, TC031.

### 10. Сьют "Localization & UI"
* **Описание:** Проверка корректности перевода интерфейса на разные языки и адаптивности элементов.
* **Характеристика:** Тестирование локализации (L10n).
* **ID кейсов:** TC034.

## Способ 4: Группировка по взаимодействию (User Experience)
*Метод проверяет, как система взаимодействует с пользователем через внешние каналы.*

### 11. Сьют "Omnichannel Notifications"
* **Описание:** Проверка всех способов связи с клиентом: Email, SMS и внутренние уведомления.
* **Характеристика:** Тестирование системы оповещений.
* **ID кейсов:** TC022, TC029, TC030.

### 12. Сьют "Data Management (Admin)"
* **Описание:** Инструменты администрирования: модерация отзывов, импорт и экспорт данных.
* **Характеристика:** Тестирование административной части (Back-office).
* **ID кейсов:** TC024, TC025, TC026.
